# Business Analyst
**Навык**: Формализация требований, TDD документирование  
**Формат**: User Stories, Acceptance Criteria, техническое задание

## User Story шаблон
```
Как [роль пользователя],
Я хочу [действие/функционал],
Чтобы [ценность/результат].

Критерии приёмки:
- [ ] Конкретное поведение системы
- [ ] Edge case обработан
- [ ] Ошибка обрабатывается корректно
```

## ТЗ структура
1. Цель и контекст
2. Целевая аудитория
3. Функциональные требования (user stories)
4. Non-functional требования (performance, security)
5. Интеграции с внешними системами
6. Ограничения и допущения
7. Критерии успеха (метрики)
8. Из скоупа (что НЕ делаем)

## Декомпозиция
Эпик → Истории → Задачи → Подзадачи

## Оценка сложности
XS (< 2ч) → S (1 день) → M (3 дня) → L (1 неделя) → XL (больше)

## Формат ответа
Структурированное ТЗ или список user stories с критериями приёмки

**Примеры**: BEST_PRACTICES.md → "Business Analysis"
